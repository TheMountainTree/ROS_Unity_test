## 核心文件变更详细说明

### 1. `ssvep_communication_node2_config.py` - Node2 静态配置模块

新增静态配置文件：

- `src/eeg_processing/eeg_processing/ssvep_communication_node2_config.py`

配置结构按功能划分为：

- `GeneralConfig`
- `UnityCommConfig`
- `TriggerForwardConfig`
- `EEGServerConfig`
- `DecodeConfig`
- `PretrainConfig`
- `ReasonerConfig`
- `SSVEPCommunicationConfig`

主要变化：

- Unity decode/pretrain 共享 `host_ip`，只保留各自端口
- trigger 转发链路独立为一组设备配置
- EEG TCP 链路独立为一组设备配置
- reasoner 的 topic、history UDP 和缩略图尺寸统一放入 `ReasonerConfig`

这样后续修改默认配置时，只需要直接编辑该文件，而不必在 Node2 内堆积大量 ROS 参数声明。

---

### 2. `SSVEP_Communication_Node2.py` - 运行时参数面瘦身

Node2 当前保留的 ROS 参数只剩：

- `run_mode`
- `reasoner_mode_enabled`
- `mock_selected_index`
- `save_dir`
- `image_dir`
- `decode_max_trials`

也就是说：

- 联调开关仍然可以在启动时通过 `--ros-args -p` 覆盖
- `mock_selected_index` 仍然可以在运行时通过 `ros2 param set` 注入
- 其他大部分默认值改为静态配置文件维护

对应启动示例：

```bash
ros2 run eeg_processing ssvep_communication_node2 --ros-args \
  -p run_mode:=decode \
  -p reasoner_mode_enabled:=true
```

对应动态选图示例：

```bash
ros2 param set /central_controller_ssvep_node2 mock_selected_index 0
```

## Pretrain Trigger 问题排查记录

用户在 pretrain 模式下执行：

```bash
ros2 run eeg_processing ssvep_communication_node2 --ros-args -p run_mode:=pretrain
```

发现 Windows 侧 `windows_com_publisher.py` 没有看到 `trigger 1/2`。

本次排查确认：

- Node2 在 pretrain 下的 trigger 发送逻辑仍然存在
  - 进入 `stim` 时发送 `trigger 1`
  - 进入 `rest` 时发送 `trigger 2`
- 默认目标地址仍为 `192.168.56.3:8888`
- Windows 侧接收脚本 `src/publisher_test/windows_com_publisher.py` 监听配置与 Node2 默认值一致
- 网络连通性（`ping 192.168.56.3`）正常

最终结论：

- 本次现象不是 Node2 重构引入的软件逻辑问题
- 原因定位为设备侧 bug

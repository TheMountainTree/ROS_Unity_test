## 核心文件变更详细说明

### 1. `publisher_test/reasoner_publish_test.py` - 24 图分组测试节点

新增测试节点 `reasoner_publish_test.py`，功能如下：

- 默认从 `~/Pictures/截图` 中按自然排序取前 24 张图片
- 切分为 4 组，每组 6 张
- 通过 `/reasoner/images` 向 `SSVEP_Communication_Node.py` 发送图片组
- 监听 `/reasoner/feedback`：

  - 收到 `cmd=request_next_group` 后发布下一组 6 张
  - 收到 `kind=history_return` 时接收 history 图片包

对应 ROS 入口：
```bash
ros2 run publisher_test reasoner_publish_test
```

并在 `src/publisher_test/setup.py` 中注册了 console entry：

### 3. 握手机制 - 解决“两个节点必须同时启动”的问题

初始版本中，`reasoner_publish_test.py` 和 `SSVEP_Communication_Node.py` 启动顺序较敏感，需要较准确的订阅时机才能让首组图片进入通信节点。

为解决这个问题，引入了显式握手：

- communication 周期发送：`cmd=ssvep_ready`
- reasoner 收到后返回：`cmd=reasoner_ready`
- reasoner 回完 `reasoner_ready` 后才发布第 1 组图片
- communication 在握手完成前不会消费 reasoner 图片批次

这样启动顺序不再重要，只要两个节点都启动，最终都会进入正常流程。

### 4. `mock_selected_index` 缓存机制 - 解决提前设参数丢失

初始实现只有在 `reasoner_wait_selection` 状态才读取参数，导致用户如果过早执行：

```bash
ros2 param set /central_controller_ssvep_node2 mock_selected_index 0
```

表面上会返回 success，但由于 trial 尚未结束，参数可能看起来没有即时生效。

改进后：

- 节点每个 timer 都轮询 `mock_selected_index`
- 一旦发现不是 `-1`，立即缓存到 `pending_mock_selection`
- 同时把 ROS 参数重置回 `-1`
- 等进入 `reasoner_wait_selection` 后再真正消费缓存值

新增日志示例：
```text
cached mock_selected_index=0, current_state=decode_stimulating
[Decode Trial 1] waiting mock_selected_index, cached=0
```

### 5. History 缩略图异常 - Unity ScrollView 间距过大

实际联调时发现，history 区出现了“两个元素之间距离特别大”的现象。

#### 原因分析

之前单独测试 `HistoryManager.cs + history_sender.py` 时，history 图片在 ROS 端已经统一缩放为 `100x100`。但新 reasoner 流程里，history 发布直接沿用了主显示图片尺寸 `640x480`。

`HistoryManager.cs` 本身并不会重新规范 history item 尺寸，只是：

- `Instantiate(iconPrefab, contentTransform, false)`
- 绑定 texture 到 `RawImage` / `Image`

因此 Unity 的 Layout Group / ContentSizeFitter 会受到输入纹理尺寸影响，导致 ScrollView 内容被拉开。

#### 修复

在 `SSVEP_Communication_Node.py` 中，history 发布前统一缩放为：

- `history_image_width = 100`
- `history_image_height = 100`

从而恢复与 `history_sender.py` 相同的输入协议。

### 6. 槽位索引与选图不一致 - reasoner 模式仍沿用随机映射

后续联调又发现：

- Unity 布局是 2 行 4 列：第一行 `0 1 2 3`，第二行 `4 5 6 7`
- `3` 是勾，`7` 是叉
- 设置 `mock_selected_index=0` 时，history 里进入的却不是左上角那张

#### 根因

虽然 reasoner 模式已经使用固定 6 张外部图片，但 decode 状态机仍然沿用了旧版逻辑中的：

```python
random.shuffle(order)
```

也就是说，屏幕上各槽位显示的是打乱后的图，而 history 写入时却按“输入组顺序”选图，二者不一致。

#### 修复

在 `reasoner_mode_enabled=true` 时关闭 shuffle，改为固定映射：

- `0 -> 第一张`
- `1 -> 第二张`
- `2 -> 第三张`
- `4 -> 第四张`
- `5 -> 第五张`
- `6 -> 第六张`

从而保证 `mock_selected_index` 与 Unity 屏幕真实槽位一致。

### 7. Decode 控制话题拆分 - `/image_seg` 不再承载命令帧

后续联调中，为了让 decode 像 pretrain 一样有独立控制通道，主链路新增了：

- 图片话题：`/image_seg`
- decode 控制话题：`/ssvep_decode_cmd`
- pretrain 控制话题：`/ssvep_train_cmd`

#### 调整内容

- `SSVEP_Communication_Node.py` 新增参数 `decode_command_topic`
- decode 的 `prepare / stim / stop / done` 不再混在 `/image_seg` 的 `cmd=` 帧里
- `SSVEP_Stimulus2.cs` 新增 `decodeCommandTopic`
- Unity 侧新增第三个订阅，用于单独接收 decode 控制命令

#### 结果

- `/image_seg` 现在只承载 6 张 decode 图片
- `/ssvep_decode_cmd` 现在只承载 `prepare / stim / stop / done`
- pretrain 继续独立走 `/ssvep_train_cmd`

### 8. Decode 停止后保留画面 - 修复 stop 后图片消失

拆分 decode 控制话题之后，实际联调时又发现：

- 闪烁结束后 ROS 已正确发送 `cmd=stop`
- 但 Unity 侧图片仍然立即消失，没有保持在屏幕上等待下一步指令

#### 根因

`SSVEP_Stimulus2.cs` 中 `stop` 仍沿用旧的收尾逻辑：

- `StopCurrentStimulationVisuals(...)` 会执行 `stimulusPanel.SetActive(false)`
- `ResetDecodeBatchState(...)` 会立即销毁当前 batch 的 6 张纹理

因此即使 decode 命令已经独立到 `/ssvep_decode_cmd`，只要 `stop` 走旧逻辑，图片仍然不会保留。

#### 修复

新增 decode 专用停止逻辑：

- `StopDecodeStimulationKeepVisuals(...)`

其行为为：

- 停止所有闪烁协程
- 清空 frame alpha
- 保持 `stimulusPanel` 激活
- 保留当前 6 张图片纹理
- 继续允许下一批图片到来后覆盖当前画面

并将以下路径改为使用该逻辑：

- decode `cmd=stop`
- decode `cmd=done`
- 新 trial 到来且 `trialId > currentTrialId` 时的收尾

## 当前 reasoner 模式交互语义

屏幕布局：

```text
0 1 2 3
4 5 6 7
```

其中：

- `3`：勾
- `7`：叉
- 动态图片槽位：`0 1 2 4 5 6`

选择语义：

- `0/1/2/4/5/6`
  - 将对应槽位图片发布到 `/history_image`
  - 发送 `cmd=request_next_group`
  - 等待 reasoner 下发下一组 6 张
- `3`
  - 将当前全部 history 图片打包回传给 reasoner
  - 当前 6 张保持不变
- `7`
  - 通过 UDP 向 Unity 发送 `delete_last`
  - 删除最后一个 history
  - 返回上一组 6 张重新选择

范围：`history_sender.py` 与 `HistoryManager.cs`

1. 已实现功能
- ROS 端可发布历史图片到 `/history_image`，并在 `header.frame_id` 写入 `hist_id=<id>`。
- Unity 端订阅 `/history_image`，支持 `bgr8/rgb8`，将图片实例化到历史列表并自动滚动到最右侧。
- ROS 端支持本地图片优先发送（目录扫描），失败时回退随机色块图。
- ROS 端对图片做统一预处理：可选垂直翻转 + 固定尺寸（默认 `100x100`），减少 Unity 布局抖动。
- 删除能力已打通：ROS 端通过独立 UDP 通道发送删除命令，Unity 端接收并执行。
  - `delete_last`
  - `clear / delete_all`
  - `delete_id`
开启测试
```
ros2 run eeg_processing history_sender_node 
```

删除命令
```
ros2 topic pub /history_control std_msgs/msg/String "{data: delete_last}" -1
ros2 topic pub /history_control std_msgs/msg/String "{data: clear}" -1
ros2 topic pub /history_control std_msgs/msg/String "{data: delete_id:3}" -1

```

1. 通信与通道
- 图像通道：ROS Topic `/history_image`。
- 控制入口：ROS Topic `/history_control`（字符串命令）。
- 删除执行链路：`history_sender.py` -> UDP `127.0.0.1:12001` -> `HistoryManager.cs`。
- 端口隔离：历史模块使用 `12001`，与 SSVEP 使用的 `9999/10000/10001` 分离，避免冲突。

3. 当前行为
- `history_sender.py` 默认最多自动发送 10 张后停止自动发送，但节点保持在线（可继续响应删除命令）。
- Unity 启动可清空 `Content` 初始子项（`clearInitialContentChildren=true`），避免初始白色占位项。

4. 已知风险/注意事项

文件：[`ssvep_processing_fbcca.py`](/home/themountaintree/workspace/ROS_Unity_test/src/eeg_processing/eeg_processing/ssvep_processing_fbcca.py)

1. 模块目标  
- 提供一套可复用的 SSVEP-FBSCCA 流水线，用于“非预训练”解码。  
- 与现有 eTRCA 流程保持接口风格一致，支持性能对比。

2. 当前已实现能力  
- `SSVEPDataLoaderFBSCCA`：加载 Nakanishi2015 数据集，输出 `X/y/meta`。  
- `SSVEPPretrainerFBSCCA`：配置 FBSCCA（滤波器组 + 参考信号），并支持 `save/load`。  
- `SSVEPDecoderFBSCCA`：执行预测 `decode(X)`。  
- `SSVEPEvaluatorFBSCCA`：单次评估、混淆矩阵、k 折交叉验证、报告打印。  
- `compare_etrca_fbscca()`：在同一数据上对比 eTRCA 与 FBSCCA。  
- `main()`：端到端测试（加载、配置、解码、评估、CV、持久化一致性、算法对比）。

3. 关键默认配置  
- 采样率 `256 Hz`，片段 `3.0 s`，偏移 `0.15 s`。  
- 通道：`["O1", "Oz", "O2", "POz"]`。  
- 滤波器组：`wp=[(6,50),(14,50),(22,50)]`，`ws=[(4,52),(12,52),(20,52)]`。  
- 谐波数 `5`，CCA 成分数 `1`。  
- 模型输出：`assets/ssvep_fbscca_model.pkl`。

4. 通信/依赖边界  
- 该文件不依赖 ROS API，可作为算法模块独立导入。  
- 依赖 `numpy/scipy/metabci.brainda`；数据加载与部分 CV 索引依赖 MetaBCI 数据与工具。  
- 启动时设置了 MNE 环境变量并 monkey patch 了 `mne.datasets.utils._do_path_update`。

5. 当前风险与注意点  
- 标签到频率映射使用 `self.freqs[int(lbl) % len(self.freqs)]`，若标签编码不连续或含异常值，可能映射错频。  
- CV 回退分支是随机切分（非严格分层）。  
- 顶层 `metabci` 导入失败会 `sys.exit(1)`，作为库被其他模块导入时不够温和。  

6. 现阶段结论  
- 模块已具备离线评估与模型持久化能力，且结构上可迁移到统一控制节点。  
- 若用于生产实时解码，建议优先加“标签-频率显式映射校验”和“更稳健的交叉验证切分”。
